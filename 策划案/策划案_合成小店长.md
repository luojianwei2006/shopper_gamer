## 1. 项目概述

### 1.1 一句话定位
玩家在一家 3D 低多边形便利店中，**点击供货 → 拖拽合成 → 上架售卖 → 挂机赚币**；店里常来固定的街坊邻居，经营会随**章节剧情**推进，过程中观看激励视频可获得加速 / 翻倍 / 额外收益，并积累**可兑换积分**，在兑换商城换取品牌卡券或实物奖励。它不是硬核生存模拟，而是一家"有故事、有熟人"的迷你人生便利店。

### 1.2 为什么是"合成 + 放置 + IAA + 边玩边赚"
- **合成**提供高频、低门槛的"爽感"核心动作（30 秒上手，每次合并都有正反馈）。
- **放置**提供"离开也在成长"的安全感，拉长会话间隔容忍度，契合移动端碎片化。
- **IAA** 是本项目主变现：激励视频把"看广告"包装成"玩家主动要的增益"，而非打扰。
- **边玩边赚**用"可兑换积分"承接用户动机——但定位为**虚拟积分兑换品牌权益**，规避直接现金提现的合规风险（详见第 7 节）。
- **完整游戏包装**：游戏不是一个功能面板的集合，而是具备"启动封面 → 载入 → 账号 → 主菜单 → 店铺世界"的完整场景流程（详见第 17 节）。让玩家"进入一个游戏"而非"打开一个工具"，这是休闲游戏留存与溢价的前提，也是前期被指出"像 App 不像游戏"的核心修复点。

### 1.3 设计假设（如与事实不符请立即纠正）
- 平均单局时长 **8–15 分钟**，日均 2–3 局。
- 目标设备：中端安卓（骁龙 6 系 / 天玑 700 级）及以上，需保证 30fps。
- 首发不强调付费内购，变现 100% 来自广告；钻石仅作为稀缺进度资源，可在后期接入极轻量 IAP。
- 美术走"低多边形 + 明亮卡通"，UI 与模型优先采用 Fab 免费资源（见《免费资源清单_Fab.md》）。

## 2. 设计支柱（Design Pillars）

> 所有后续设计决策都须对照这 6 条。任何与一个以上支柱冲突的功能，默认砍掉。

| # | 支柱 | 玩家体验目标 | 不可妥协的检验标准 |
|---|------|--------------|--------------------|
| P1 | **爽感合成** | 每次拖拽合并都"juicy"：有特效、有音效、有震动、有成长数字 | 合并瞬间必须有 ≥2 种反馈；单次合并心流不被弹窗打断 |
| P2 | **放置成长** | 关掉游戏也在变强，回来有"惊喜包裹" | 离线收益可见、可领；离线越久收益边际递减但永不为 0 |
| P3 | **边玩边赚** | 积分"看得见地"累积，兑换目标触手可及 | 任意会话结束都能看到积分增长；兑换门槛阶梯可达成 |
| P4 | **低门槛深上限** | 3 秒内懂玩，但存在优化空间（合成链规划、订单取舍） | 首次操作零失败；高级玩家有"效率最优解"可追求 |
| P5 | **长效留存** | 每天回来都有新东西：签到、任务、赛季、社交攀比 | D1/D7/D30 留存有设计抓手；无"断签即崩"的惩罚焦虑 |
| P6 | **有温度的陪伴感**（v1.4 新增） | 店铺是"我的店"，顾客是"认识的人"，经营有故事在推进 | 任意老玩家能叫出 ≥2 个常驻顾客名字；回游戏能看到"上次发生了什么"的叙事延续 |

## 3. 核心循环（Core Loop）

### 3.1 Moment-to-Moment（0–30 秒）
- **Action**：点击「供货机」产出 1 个 Lv1 商品；拖拽一个商品到同类商品上触发**合成**。
- **Feedback**：合并瞬间播放缩放弹跳 + 粒子迸发 + "叮"音效 + 轻微震动（移动端 Haptic）；升阶数字飘字。
- **Reward**：商品升 1 阶（价值指数上升）；完成首次合并即解锁"上架"提示。

### 3.2 Session Loop（5–30 分钟）
- **Goal**：把高价值商品上架货架 → 顾客自动购买产币 → 接「订单板」订单赚大额金币 + 积分 → 看激励视频**翻倍**订单奖励。
- **Tension**：货架格位有限（资源压力）+ 订单限时（时间压力）；是否用广告换效率是最频繁决策。
- **Resolution**：完成订单 → 金币/积分入账 → 解锁下一产业链或扩建；失败仅丢失该笔订单奖励，无惩罚（保留 P4 零挫败感）。

### 3.3 Long-Term Loop（小时–周）
- **Progression**：店铺等级 → 解锁新产业链（果蔬→烘焙→饮品→熟食→日用）→ 货架扩建 → 装饰评分（影响顾客数）。
- **Retention Hook**：每日签到（连签递增）、每周任务、赛季通行证、好友店铺拜访（轻社交攀比，不强制）。

### 3.4 Fun Hypothesis（核心乐趣假设）
> **"把普通东西合成出更值钱的东西"这件事本身必须好玩。** 这是整个游戏成立的前提。
> 验证方法：在没有任何经济/广告系统的情况下，单测"纯合成手感"是否让人想继续点。若合成不爽，后面全废。

## 4. 玩法机制规格（Mechanic Specifications）

### 4.1 机制：合成（Merge）
- **Purpose**：核心动词，提供高频爽感与进度感（P1/P4）。
- **Player Fantasy**："我手里的东西越变越值钱"。
- **Input**：点击供货机产出 Lv1；**长按拖拽至同类同级物品释放**（**设计已锁定：合成手势 = 拖拽**，不采用纯点击合成；已在原型②验证拖拽手感与磁吸高亮）。
- **Output**：两个同级合成一个高一级物品；占用格位减少、单体价值 ×`MergeValueMult`[PLACEHOLDER]。
- **Success Condition**：拖拽有磁吸高亮，释放 0.1s 内出合并特效；同帧多合并不卡顿。
- **Failure State**：误拖到非同类 → 物品弹回原位，无惩罚；格位满时合成结果无处放置 → 提示"扩建货架"。
- **Edge Cases**：
  - 三个同类相邻同时拖拽？→ 仅取前两个合成，第三个保持。
  - 满级物品再合成？→ 满级不可合成，提示"已满级，可上架/出售"。
- **Tuning Levers**：供货机冷却、合成链长度、每阶价值倍率、最大阶数。
- **Dependencies**：供货机、货架、订单板、经济系统。

### 4.2 机制：放置售卖（Idle / Shelf Auto-sell）
- **Purpose**：离线/挂机也成长，降低"不玩就落后"的焦虑（P2）。
- **Player Fantasy**："我的店在我睡觉时也在赚钱"。
- **Input**：将商品拖到货架格 → 自动以 `ShelfPrice` 持续售出。
- **Output**：持续产金币；高价值商品售出触发顾客小动画。
- **Success Condition**：挂机 1 分钟肉眼可见金币增长；离线收益有上限且边际递减。
- **Failure State**：货架全空 → 收益为 0，但无惩罚，引导回合成循环。
- **Edge Cases**：离线超过 8 小时[PLACEHOLDER] → 收益封顶（防通胀，见第 6 节）。
- **Tuning Levers**：售出间隔、单位售价、离线收益上限、装饰评分加成。
- **Dependencies**：经济系统、装饰系统、通胀监测。

### 4.3 机制：订单板（Order Board）
- **Purpose**：提供"目标感 + 大额奖励 + 广告触点"（P3/P4）。
- **Player Fantasy**："顾客点名要我的招牌商品，我赚到了"。
- **Input**：订单要求"N 个 LvX 商品"；玩家凑齐点击交付。
- **Output**：金币 + 积分（`OrderPointReward`[PLACEHOLDER]）；限时倒计时。
- **Success Condition**：订单交付流畅；交付后即时给"双倍？看视频"选项。
- **Failure State**：超时未完成 → 订单消失，无惩罚，刷新新订单。
- **Edge Cases**：交付瞬间玩家金币为负？→ 不可能（订单奖励只增不减）。
- **Tuning Levers**：订单刷新频率、需求数量、奖励曲线、限时长度。
- **Dependencies**：合成、货架、广告系统、经济系统。

### 4.4 机制：广告激励（Ad Boost）
- **Purpose**：把"看广告"变成玩家主动追求的增益，而非打扰（IAA 核心）。
- **Player Fantasy**："我看个短视频，就能省时间/多拿奖励，划算"。
- **Input**：玩家主动点击带"视频"图标的增益按钮。
- **Output**：触发激励视频；完成播放后生效（双倍奖励 / 加速产出 / +1 合成格 / 离线收益翻倍领取）。
- **Success Condition**：广告播放完成才发奖；断播不发（防作弊与平台合规）。
- **Failure State**：广告加载失败 / 无填充 → 优雅降级为"稍后再试"，不卡流程。
- **Edge Cases**：
  - 玩家在广告播放中切后台？→ 不计完成，无奖励。
  - 单日广告次数触顶？→ 按钮灰显并提示"今日已领完，明日再来"。
- **Tuning Levers**：各增益数值、每日广告上限（`DailyAdCap`[PLACEHOLDER]）、冷却。
- **Dependencies**：广告 SDK（见第 5 节）、经济系统、订单板。

### 4.5 机制：积分兑换（Redemption）
- **Purpose**：承接"边玩边赚"动机，形成长线目标（P3）。
- **Player Fantasy**："我玩游戏攒的积分，真能换到东西"。
- **Input**：玩家在兑换商城选择奖品 → 扣除积分。
- **Output**：发放卡券码 / 实物发货（对接第三方发放服务）。
- **Success Condition**：兑换后即时到账提示 + 历史记录可查。
- **Failure State**：库存不足 → 提示补货中，积分不退（或退，按运营策略）。
- **Edge Cases**：未成年人兑换？→ 接入年龄/实名校验，限制高价值实物（合规见第 7 节）。
- **Tuning Levers**：积分汇率、各奖品门槛、库存、抽奖概率（若含）。
- **Dependencies**：积分经济、合规校验、第三方发放。

## 5. IAA 广告系统设计

### 5.1 广告位总览
| 类型 | 触发点 | 玩家主动性 | 频率上限 |
|------|--------|------------|----------|
| **Rewarded Video（激励视频）** | **8 类触发点**（见下，主动点击，均经服务端 S2S 校验发奖） | 主动（玩家点） | `DailyAdCap`[PLACEHOLDER] 次/日 |

**激励视频 8 类触发点（主动，玩家点）**：

| # | 触发点 | 增益内容 | 设计意图 / 对应系统 |
|---|--------|----------|---------------------|
| 1 | **订单双倍** | 当前订单奖励金币 + 积分 ×2 | 核心转化点，P1 爽感叠加 |
| 2 | **生产加速** | 供货机/合成临时提速 | 缓解产能瓶颈（§16 厨师） |
| 3 | **+1 合成格** | 临时增加货架合成位 | 早期扩张爽点 |
| 4 | **每日宝箱** | 固定随机资源（金币/钻石/积分） | 日活钩子（P5） |
| 5 | **离线收益翻倍领取** | 离线收益 ×2 领取 | 回归惊喜放大（P2） |
| 6 | **离线收益领取** | 单纯领取（无翻倍也设视频点，给"想快点领"的玩家） | 扩触点，提升人均观看（§4 通胀监测参考） |
| 7 | **顾客加急配送** | 指定顾客订单立即完成 + 好感 + | 叙事与经营联动（§15.3） |
| 8 | **信誉救场 / 产能救场** | 临时 +1 厨师 或 加速合成，化解怒气爆表风险 | 防信誉崩盘（§16.3/§16.5） |
| 9 | **章节卡点加速** | 卡住的章节目标进度 +X% | 叙事推进保护（§15.4） |

> 注：原"生产加速"与"产能救场"在 §16 语境下合并为同类产能增益，此处拆为 2 个独立触点以便埋点区分转化。触发点编号 1–9 实际为 9 类，统计口径以埋点为准（`DailyAdCap`[PLACEHOLDER] 为日总上限，不因触点多而放宽单触点频率）。
| **Interstitial（插屏）** | 完成里程碑订单、进入新"营业日"、自然节点 | 被动 | 每会话 ≤2 次，两次间隔 ≥90s[PLACEHOLDER] |
| **Banner（横幅）** | 店铺主界面底部常驻 | 被动（可收起） | 单屏 1 条，非侵入式 |
| **Offerwall（可选）** | 高价值积分任务（调研/下载试玩） | 主动 | 运营配置，低频 |

### 5.2 设计原则
1. **激励视频必须"玩家想要"**：增益价值 > 看 15–30s 视频的成本，才点。绝不强制。
2. **插屏不过度**：频率上限与冷却是为了保护留存与商店评分——打扰过多会直接拉低 LTV。
3. **Banner 可收起**：提供一个"✕"关闭，降低负面体感。
4. **加载失败优雅降级**：任何广告失败都不能阻断核心循环。
5. **SDK 接入**：推荐 LevelPlay（ironSource）或 AdMob 聚合，统一管理 Waterfall / Bidding，提升 eCPM。

### 5.3 广告频率"破防"预警（定义 Broken）
- 单会话插屏 >3 次，或激励视频按钮遮挡核心操作 → **设计失败**，立即回滚频率配置。
- 玩家因广告放弃交付订单比例 > `AbandonThreshold`[PLACEHOLDER] → 触发平衡复盘。

## 6. 经济系统（概要，详见《经济系统平衡表.md》）

### 6.1 三套货币
| 货币 | 性质 | 主要来源 | 主要消耗 | 设计意图 |
|------|------|----------|----------|----------|
| **金币 Coin** | 软货币 | 售卖、订单、挂机 | 扩建货架、解锁产业链、装饰 | 日常进度燃料 |
| **钻石 Gem** | 稀缺资源 | 成就、赛季、极少 IAP（后期） | 高级加速、稀有装饰 | 长线稀缺感（非变现主轴） |
| **积分 Point** | 可兑换 | 订单分成、广告里程碑、任务 | 兑换商城 | "边玩边赚"载体 |

### 6.2 经济健康标准
- **可解性**：所有路径无死循环、无无限刷（订单/广告增益设冷却与上限）。
- **通胀监测指标**：`人均日产出积分`，超过阈值 `InflationThreshold`[PLACEHOLDER] → 触发汇率/门槛平衡 pass。
- **离线收益封顶**：`OfflineCapHours`[PLACEHOLDER] 小时后收益不再累积，防长期挂机通胀。

## 7. 兑换体系与合规

### 7.1 兑换物分层（由低到高，保证"触手可及"）
1. **品牌优惠券**（如 ¥5 话费券 / 电商无门槛券）——低门槛，拉首次兑换率。
2. **中等卡券**（如 ¥20 视频会员 / 外卖券）——中门槛，承接周活。

## 14. 后端网络架构（Socket ↔ Java 服务器）

### 14.1 为什么需要服务器（设计前提）
本项目"边玩边赚 + 可兑换积分"是**作弊高敏感**场景。若积分/订单由客户端自管，玩家可通过改内存、伪造广告回调无限刷分，经济体系直接崩坏。因此 Java 服务器必须是**权威方（server-authoritative）**：
- 积分余额、订单奖励、广告激励发放**以服务端为准**；
- 广告激励必须走广告平台 **S2S（server-to-server）回调校验**，客户端仅上报"我看完了"，服务端校验后才发奖；
- 云存档（进度、店铺等级）服务端落库，防丢档与跨设备。

### 14.2 推荐方案（决策）：混合架构 HTTP REST + TCP 长连接
> **无状态、请求-响应型、可缓存、可重试的操作走 HTTP REST；需要服务端主动推送、低延迟、长生命周期权威状态的走 TCP 长连接。**

**设计原则**：登录、取数据、存档这类操作天然是"一问一答、无状态、可重试"，用 HTTP 最省心（负载均衡无状态、CDN 可缓存、调试直观）；而"激励视频发奖、积分实时到账、订单刷新推送"需要服务端**主动**告诉客户端，长连接才合适，避免客户端轮询空耗流量。

| 操作 | 通道 | 序列化 | 理由 |
|------|------|--------|------|
| 账号登录 / 换取会话令牌 | **HTTP** `POST /auth/login` | JSON | 一次性、无状态；返回 `token` 用于后续两条通道鉴权 |
| 拉取经济配置 / 静态表（经济数值、兑换目录、活动配置） | **HTTP** `GET /config/*` | JSON | 低频、可 CDN 缓存、可读好调试 |
| 云存档上传 / 下载（店铺进度、关卡、设置） | **HTTP** `PUT/GET /save` | JSON | 低频、幂等、断点续传友好 |
| 兑换申请（积分 → 实物/卡券） | **HTTP** `POST /redeem` | JSON | 事务型、需订单回执、可审计、需风控限频 |
| 排行榜 / 好友 / 活动查询 | **HTTP** `GET /rank` | JSON | 读多写少、可缓存 |
| 激励视频 S2S 校验 + 积分发放 | **TCP** | Protobuf | 实时、需服务端确认广告平台回调后推送 `S2C_PointGrant` |
| 订单刷新 / 在线事件 / 实时活动推送 | **TCP** | Protobuf | 服务端主动 push，避免轮询 |
| 重连后权威状态校正（余额/待处理订单） | **TCP** | Protobuf | 长连接内 `C2S_PullState` 拉取 |

**协议选型**
- **HTTP**：HTTPS + JSON。可读、好调试、CDN 缓存友好、无状态负载均衡；UE5 用内置 `FHttpModule` / `FHttpRequest` 异步调用，主线程回调，**无需自建线程**。
- **TCP 长连接**：Protobuf + 4 字节大端长度头分包。紧凑、强类型、长连接省流量；Java 端 Netty `LengthFieldBasedFrameDecoder`。

**Java 服务端（职责分离）**
- **Spring Boot**：提供 REST（登录 / 取数 / 存档 / 兑换 / 排行榜），无状态、水平扩展容易。
- **Netty（独立）**：提供 TCP 长连接（实时事件），NIO 事件循环单机数万连接。
- 两者**共享同一用户/积分服务与数据库**，`token` 统一签发校验；兑换与积分发放落同一事务，避免双通道数据不一致。

**UE5 客户端**
- HTTP：`FHttpModule` / `FHttpRequest`（异步，主线程回调）。
- TCP：`FSocket`（`Sockets`/`Networking`）+ `FRunnable` 收线程 + `AsyncTask(GameThread)` 派发（§14.4）。

**鉴权流（关键）**：`HTTP /auth/login` → 拿到 `token` → TCP 建连时携带 `token` 做握手鉴权 → 鉴权通过才启用实时事件通道。两条通道都靠同一 `token` 校验，杜绝"跳过登录直接连 Socket 刷分"。

**备选**：若未来需 Web/小游戏互通，TCP 长连接可升级为 **WebSocket**（UE5 `IWebSocket` + Java Netty WS）；休闲游戏消息低频，WS 开销可接受。HTTP REST 层保持不变。

### 14.3 接口与协议示例
**HTTP（REST，JSON，HTTPS）**
- `POST /auth/login` { deviceId, channel } → { token, uid, expireAt }
- `GET /config/economy` → { goldRate, pointRate, redeemCatalog... }（可 CDN 缓存）
- `GET /save?slot=0` → { saveData } ； `PUT /save` { saveData }（云存档）
- `POST /redeem` { token, rewardId } → { ok, orderId, eta }
- `GET /rank?type=daily` → { entries[] }

**TCP（Protobuf，长连接，token 握手后启用）**
- `C2S_AdReward`{ adId, placement, signature } → 服务端 S2S 校验广告平台 → `S2C_PointGrant`{ delta, balance }
- `C2S_PullState`{} → `S2C_State`{ pointBalance, pendingOrders }
- `S2C_Push`{ type, payload }（订单刷新、积分活动、维护公告）

> 旧版 `C2S_Login` / `S2C_LoginAck` 已弃用：登录改走 HTTP，TCP 仅保留 `token` 握手鉴权，不再承担登录取数职责。

### 14.4 客户端线程与可靠性
- **HTTP**：`FHttpRequest` 异步发送，主线程 `OnProcessRequestComplete` 回调处理，**无阻塞风险**；失败按状态码重试（429/5xx 指数退避），存档类请求做本地队列防丢。
- **TCP 收线程**：`FRunnable::Run` 中循环 `Recv`，按长度头拼完整包入线程安全队列。
- **派发**：游戏线程取出队列包，路由到对应 Handler（UMG 刷新 / 事件分发）。
- **心跳**：每 `HeartbeatInterval`[PLACEHOLDER] 发 ping，服务端 `HeartbeatTimeout`[PLACEHOLDER] 内无响应则断开。
- **重连**：断线指数退避重连；重连后 `C2S_PullState` 拉取权威状态校正本地（防本地作弊残留）。
- **加密**：HTTPS 已全程加密；TCP payload 生产环境做轻量 HMAC/Anti-Cheat 校验，防中间人改包。

### 14.5 职责边界（设计不越界）
- 本协议/线程模型为**设计层建议**：HTTP REST 与 TCP 长连接的具体 C++ 类结构、Spring Boot Controller、Netty Pipeline、`.proto` 定义在工程侧落地（轻量底座见《免费UE5框架参考.md》：Top Down 模板 + TCP-Unreal / UETcpSocketPlugin 插件；HTTP 用 UE5 内置 `FHttpModule`，零额外插件）。
- 服务器权威是"边玩边赚"的合规与可持续前提，**不可省略**；无论走 HTTP 还是 TCP，积分余额、订单奖励、广告激励发放均以服务端为准，否则积分经济不可信。

## 15. 剧情化、3D 店铺场景与轻量模拟人生

> 设计方向（已与用户确认）：**轻量陪伴感 + 章节任务链 + 经营舞台**。
> 即：不做生存模拟（吃喝睡），不强制主线，3D 店铺是沉浸式背景而非主玩空间。新增设计支柱见 P6。

### 15.1 设计意图（为什么加这一层）
- 把"合成经营"从纯数值循环拔高为**有温度的迷你人生**，提升代入感、留存与兑换意愿（对照 P3/P5/P6）。
- 取舍原则：**只做"让人想回来的陪伴感"，不做硬核生存需求**——避免把休闲 IAA 做成重度模拟，控制工作量与移动端性能。
- 这是 P1「爽感合成」之上的"第二层钩子"：合成负责即时爽，剧情与熟人负责长期黏。

### 15.2 3D 店铺场景（全 3D 卡通小人经营舞台）
- **定位（真 3D · 45° 室内 · 卡通）**：**全 3D 低多边形场景**，采用卡通小人（Fab Low Poly People / Cartoon Characters 类资源）作为店长、店员、厨师、顾客模型；**相机为室内 45° 俯视**（类《模拟人生》/《双点医院》系列），玩家看见店内货架、收银台、**入口门、出口门、店员座位排与厨师工位**（相机与"黑色阴影"光照揭示机制详见第 17.2 节）。
- **活着的店（经营可视化）**：这一层不仅是氛围，还**承载运营模拟**（详见第 16 节）——顾客从**入口门**进、走到**店员座位**落座生成订单、对应商品由**厨师工位**合成完成、顾客从**出口门**离开，全程带"往前移动"动画，店在"真运转"。
- **上货有空间感**：合成产物从"供货机"动画送出落到货架格。
- **与合成 UI 板的关系**：高频手动操作（如临时拖拽、广告弹窗）仍用 2D UI 板浮于场景之上；但**招募店员这类"点座位"操作直接在 3D 中点选座位**——为保移动端点击精度，点座位后弹出 UMG 确认气泡（显示花费 100 金币 + 确认/取消），避免误触小模型（对照 P4 低门槛）。
- **性能策略**（对照 §1.3 中端机 30fps 硬指标）：
  - 顾客/店员/厨师 NPC 用 Instanced Static Mesh + 简单状态机，同屏上限 `MaxNPC`[PLACEHOLDER]（建议 6–12，含店员/厨师）。
  - 低端机降配：关动态阴影、降 NPC 数量、开纹理流送（见《免费UE5框架参考.md》优化项）。
  - 场景与角色资产来自 Fab 免费 Low Poly Scenario Pack / Low Poly People Pack / Low Poly Ultimate Pack（见《免费资源清单_Fab.md》）。

### 15.3 轻量模拟人生系统（陪伴感，非生存）
- **店长形象**：玩家拥有一个可自定义店长（Fab 低多边形角色，如 Low Poly People Pack），可在"我的小屋"换装 / 摆放装饰。
  - *设计杠杆（Endowment Effect）*：让玩家**先投入时间/积分装扮小屋**，沉没成本抬高后续留存（对照行为经济学模块）。
- **常驻顾客 NPC**：每位有**名字、固定偏好商品、好感度**。
  - 好感度来源：完成其订单、看广告为其"加急配送"、赠送小礼。
  - 好感度收益（设计杠杆，须进《经济系统平衡表》建模）：
    - Lv1→Lv2：该顾客下单频率 +`AffinityFreqBonus`[PLACEHOLDER]
    - 满好感：解锁该顾客专属剧情 + 一次性大额积分奖励（可作**首兑助推**，见 §7）
  - *不做*：顾客不吃喝睡、无健康/情绪崩溃——保持轻量，守住"休闲"边界。
- **小镇/邻里（轻社交）**：可选拜访好友店铺（只读展示），攀比装饰评分与店铺等级，不强制（对照 P5）。

### 15.4 章节任务链剧情
- **结构**：随**店铺等级**解锁章节（Chapter 1…N）；每章 = 一段主线小故事 + 一串经营目标 + 章节奖励。
- **叙事驱动经营**：剧情目标与经营动作绑定（如"帮阿婆准备寿宴蛋糕"→ 推进烘焙产业链 + 完成 3 笔蛋糕订单）。
- **顾客小故事**：每位常驻顾客带支线，边经营边通过对话气泡 / 信件解锁，形成"我认识这条街上的人"的陪伴感。
- **示例章节（纸面草案，文案待细化）**：
  - **第一章·开张第一天**：顾客"小满"来买早餐，教会玩家合成面包；目标 = 升到 Lv2 店铺。奖励 = 解锁饮品产业链 + 50★。
  - **第二章·街角的熟客**：常客"老周"成为首位满好感顾客，引出他的早餐摊故事；目标 = 完成 10 单 + 装饰评分达 30。奖励 = 小屋装饰×1 + 100★。
- **节奏控制**：章节对话**可跳过**，且**不在合成高潮时强制弹出**（保护 P1 爽感）；改为"信件收件箱"异步送达，玩家主动查看。

### 15.5 与现有系统关系
- **经济**：好感度增益、章节奖励、满好感积分均产出金币/积分 → **已在《经济系统平衡表》§8 建模**（好感频率加成 `AffinityFreqBonus`、章节积分 `ChapterPointReward`、满好感首兑助推 `AffinityFirstRedeemBoost`），防通胀（对照第 6 节通胀监测）。
- **IAA**：顾客"加急配送"、章节卡点可用激励视频加速（对照第 8 节 IAA）。
- **兑换**：满好感一次性积分奖励设计为"首兑助推"，让小鱼玩家更快触达首个兑换门槛（对照第 7 节）。
- **后端**：顾客好感度、章节进度、小屋装饰均为**服务端存档**（HTTP `/save`），防作弊与跨设备（对照第 14 节）。

### 15.6 风险与回滚
- **Scope creep（最大风险）**：模拟人生最易膨胀。回滚线 = 若 NPC 需要寻路 AI / 日程系统，立即砍到"固定路线 + 好感度数字"，不影响核心。
- **性能**：3D 场景 + NPC 是中端机主要风险点；以 §1.3 的 30fps 为硬指标，超预算则降 `MaxNPC` 与特效。
- **叙事打断节奏**：章节弹窗若影响合成心流，改为"信件收件箱"异步送达（已在 §15.4 预设）。
- **定义"坏"（上线前对齐）**：
  - 章节完成率 < 40% → 叙事过重，削减。
  - "跳过剧情"占比 > 70% → 玩家反感强制叙事，转为纯环境彩蛋。
  - 3D 场景导致中端机 fps < 30 持续 → 降配或改为 2.5D 伪 3D。

## 16. 店铺运营模拟层：店员 / 厨师 / 顾客流转 / 信誉

> 设计方向（已与用户确认）：全 3D 卡通小人场景；店员/厨师构成经营"产能与吞吐"骨架；顾客从入口门进 → 到店员座位落座生成订单 → 对应商品由厨师合成完成 → 订单完成 → 从出口门离开；等待过久积累怒气，怒气爆表扣信誉。
> 本层把"合成"从纯数值循环升级为**有吞吐瓶颈的经营管理模拟**，制造"店员拉单量 vs 厨师产能"的策略张力，承接 P4「低门槛深上限」与 P6「有温度的陪伴感」。

### 16.1 设计意图
- 给经营一个**可感知的瓶颈**：玩家直观看到"单子堆了、厨师忙不过来、顾客在冒火"，从而决策"招店员还是招厨师"。
- 核心策略张力：店员↑ → 顾客列数↑ → 订单量↑；但厨师不足 → 产出跟不上 → 等待↑ → 怒气↑ → 信誉↓。故**店员与厨师须按比例扩张**。
- 信誉为长期**软惩罚**（非硬失败，守 P4 零挫败）：信誉低 → 顾客变少/奖励打折，但游戏不卡死，补产能后自愈。

### 16.2 核心概念与关系
| 概念 | 作用 | 上限来源 |
|------|------|----------|
| 厨师 Chef | 决定**合成产能**：每厨师 = 1 条并行合成管线；厨师越多单位时间产出越多 | 店铺等级（§16.4） |
| 店员 Clerk | 决定**订单吞吐列数**：坐座位接待顾客；店员数 = 顾客队列列数（1 店员=1 列，5 店员=5 列） | 座位排上限 5，受等级约束 |
| 顾客 Customer | 入口进 → 店员座位落座 → 生成 1 笔订单；对应商品合成完成 → 订单完成 → 出口离开 | 同屏 `MaxNPC`（§15.2） |
| 信誉 Reputation | 全局店铺声望 0–100；顾客怒气爆表时扣减；影响顾客到店率与订单奖励 | 初始 100，下限 0 |

**流转关系**：入口门 →（按列分流）店员座位 → 落座生成订单 → 进合成队列 → 厨师合成对应商品 → 订单完成 → 顾客走向出口门离开 → 信誉/好感结算。

### 16.3 机制规格

#### 机制 A：厨师与合成产能
- **Purpose**：把"合成数量"从手动拖拽扩展为"产能"概念，给经营规模感。
- **Player Fantasy**：后厨越强大，出货越快，顾客越满意。
- **Input**：招募厨师（消耗 `ChefCost`[PLACEHOLDER] 金币 / 等级解锁）；厨师自动持续合成，无需手动。
- **Output**：单位时间可完成合成数 = `厨师数 × (1 / 单件合成耗时 SynthTime)`。
- **Success Condition**：订单产生速率 ≤ 合成产能，顾客等待不超阈值，怒气不涨。
- **Failure State**：厨师不足 → 合成队列积压 → 等待超 60s → 怒气涨。
- **Edge Cases**：
  - 厨师数 = 0（不应发生，首店保底 1 厨师）：若发生则订单全积压触发怒气 → 需保底逻辑：开局至少 1 厨师。
  - 高阶商品合成更慢（`SynthTime × TierMult`[PLACEHOLDER]）：高价值单更占产能，倒逼扩厨师。
- **Tuning Levers**：`ChefCost`、`SynthTime`、`TierMult`、厨师等级上限。
- **Dependencies**：§3 合成核心、§15.2 厨师工位、§14 存档。

#### 机制 B：店员招募与顾客队列
- **Purpose**：店员是"接单入口"，决定同时接几路顾客。
- **Player Fantasy**：扩座位加店员，店越热闹越像样。
- **Input**：点击**空座位** → 消耗 **`100 金币`（固定）** → 该座位生成 1 名店员 → 顾客队列**列数 +1**（原 1 列变 2 列；满 5 店员 = 5 列）。
- **Output**：顾客队列列数 = 店员数；每列独立排队，入口门按列分流顾客（带"往前移动"动画）。
- **Success Condition**：店员数 ≥ 同时进店顾客所需列数，无顾客在入口长滞留。
- **Failure State**：店员满 5 仍不够（顾客暴增）→ 入口溢出，新顾客门口等待（计入怒气计时）→ 需升级店铺扩上限（§16.4）。
- **Edge Cases**：
  - 座位全满（5 店员）：不可再招募，提示"升级店铺解锁更多座位"。
  - 金币不足 100：座位置灰/不可点，Toast 提示。
  - 点击已有店员的座位：无操作（或弹该店员信息）。
- **Tuning Levers**：`RecruitClerkCost = 100`（用户指定，固定）、座位排上限 5、店员等级上限。
- **Dependencies**：§16.4 等级上限、经济表 §7.1（金币消耗）、§15.2 场景座位。

#### 机制 C：顾客流转（Customer Flow）
- **Purpose**：把"订单"具象为"一个活人从进门到离开"，给经营空间感与节奏。
- **Player Fantasy**：看店里人来人往，单子被一个个接走、做完、送走。
- **Input**：入口门按 `SpawnRate`[PLACEHOLDER] 生成顾客；顾客自动走向**空店员座位**（带"往前移动"动画）。
- **Output**：顾客落座 → 生成 1 笔订单（指定所需商品与阶数）→ 进合成队列；对应商品合成完成 → 订单完成 → 顾客走向**出口门**离开（离开动画）。
- **Success Condition**：进门 → 落座 → 完成 → 离开 全链路顺滑，无卡死。
- **Failure State**：无空座位 → 顾客入口排队；无对应产能 → 落座后订单久未完成 → 怒气涨。
- **Edge Cases**：
  - 同时进店顾客数 > 店员列数：多余顾客在入口排"虚拟队"，按列轮流进入。
  - 所需商品产业链未解锁：该订单不生成 / 标记"暂不可接"，避免死锁。
  - 重连/离线：流转状态服务端存档（§14），回归校正。
- **Tuning Levers**：`SpawnRate`、单顾客移动速度、出口动画时长。
- **Dependencies**：机制 A/B、§15.2 场景门与座位、§14 存档。

#### 机制 D：怒气值与信誉值
- **Purpose**：把"等待过久"转化为可量化惩罚，且不硬卡死游戏（守 P4）。
- **Player Fantasy**：别让顾客等急——看到他冒火了，得赶紧补产能。
- **Input**：顾客**落座后**开始计时；等待 > `WaitThreshold = 60s`（固定）→ 每 `10s` 怒气 +`AngerPerTick`[PLACEHOLDER]（默认 10）。
- **Output**：
  - 怒气 ≥ `AngerMax = 100`（固定）→ 开始扣**信誉值**：每 10s 信誉 -`RepPenalty`[PLACEHOLDER]（默认 1）。
  - 订单在怒气爆表前完成 → 顾客满意离开，可能 +好感/小费（§15.3）。
- **Success Condition**：多数顾客 60s 内完成，怒气不涨，信誉稳定 100。
- **Failure State**：长期产能不足 → 信誉持续下跌。
- **Edge Cases**：
  - 信誉 = 0：不"游戏结束"，进入"低信誉态"：顾客到店率 -`LowRepSpawnPenalty`[PLACEHOLDER]、订单奖励 ×`LowRepRewardMult`[PLACEHOLDER]（如 0.7），形成软惩罚且玩家补产能后自愈。
  - 信誉回升：完成满意订单缓慢 +信誉，或随时间自然恢复 `RepRegen`[PLACEHOLDER]。
- **Tuning Levers**：`WaitThreshold=60`、`AngerPerTick`、`AngerMax=100`、`RepPenalty`、`RepRegen`、`LowRep*` 系列。
- **Dependencies**：机制 A/B/C、§15.3 好感、§14 存档（信誉防作弊）。

### 16.4 等级 → 上限映射（招募天花板）
| 店铺等级 | 店员上限（座位排） | 厨师上限 |
|----------|--------------------|----------|
| Lv1 | 1（开局 1 列） | 1（保底） |
| Lv2 | 2 | 1 |
| Lv3 | 3 | 2 |
| Lv4 | 4 | 2 |
| Lv5 | 5 | 3 |
| Lv6+ | 5（封顶，靠品质而非数量） | +1 / 2 级 [PLACEHOLDER] |

> 数值为草案，`[PLACEHOLDER]` 需按经济表 §7 与纸面模拟校准；原则：店员/厨师上限随等级线性但封顶，避免后期无限堆叠导致经济失控。

### 16.5 与现有系统关系
- **经济**：招募店员消耗 100 金币（主 sink 之一，见经济表 §7.1）；厨师/店员上限扩张靠等级，间接消耗升级资源。
- **核心循环**：把 §3 的"订单"从抽象列表变成 3D 流转；合成从手动拖拽升级为"产能驱动"（厨师）。
- **剧情/陪伴**：常驻顾客（§15.3）也走此流转；满好感顾客离店给额外奖励。
- **IAA**：产能不足时可看广告"临时 +1 厨师 / 加速合成"（对照 §8）；信誉低时广告救场。
- **后端**：店员/厨师配置、信誉、怒气态全部服务端存档（§14 HTTP `/save`），防本地改内存刷产能。

### 16.6 风险与回滚 / 定义"坏"
- **性能**：NPC + 座位 + 厨师动画同屏，是中端机风险；以 §1.3 的 30fps 为硬指标，超预算降 `MaxNPC` 与动画细节。
- **复杂度**：招募/产能/怒气/信誉四系统耦合，易让休闲玩家迷糊。回滚线 = 若新手引导期（前 3 关）完单率 < `OnboardCompleteRate`[PLACEHOLDER]，简化为"只显示订单数 vs 产能进度条"，隐藏单列/多列细节。
- **定义"坏"**：
  - 小鱼玩家因产能不足导致信誉长期 < 50 → 惩罚过狠，调 `RepPenalty` / 加保底产能。
  - 店员/厨师扩张对留存无提升 → 该层过浅，加"厨师专精 / 店员技能"深度。
  - 信誉系统被玩家无视（不影响行为）→ 反馈不够明显，强化怒气/信誉的视觉（冒火表情、信誉条）。

## 17. 场景架构与游戏流程（去 App 感）

> 问题：此前只有"店铺实景"一堆功能面板，缺少游戏应有的开场、载入、账号、主菜单，读起来像工具 App 而非游戏。本节补齐**完整场景流程**，让玩家"进入一个游戏世界"。

### 17.1 五幕场景流程（Screen Flow）

| # | 场景 | 职责 | 关键交互 | 接入点 |
|---|------|------|----------|--------|
| 1 | **启动封面 Splash** | 品牌第一印象：工作室 LOGO + 游戏名《合成小店长》 | 点击 / 任意处 → Loading | 冷启动首屏 |
| 2 | **Loading 资源** | 异步加载 3D 资产 / 配置，进度可读 | 进度条 + 小贴士；完成 → 账号 | 资源流式加载（见 §17.3） |
| 3 | **账号 Account** | 登陆 / 注册 / 游客 | 三模式切换；HTTP `/auth/login` 拿 token → 进入菜单 | 对照 §14 HTTP REST |
| 4 | **游戏菜单 Menu** | 游戏门户：开始游戏 / 游戏说明 / 退出游戏 | 开始 → 店铺实景；说明 → 玩法说明；退出 → 平台退出 | 游戏的"大厅" |
| 5 | **店铺实景 Shop（世界）** | **唯一游戏内场景**，承载全部玩法 | 45° 真 3D 视角 + HUD + 底部功能坞 + 设置按钮（见 §17.2） | 详见 §15 / §16 |

> 设计原则：第 1–4 幕是"游戏外壳"，第 5 幕是"游戏内核"。外壳越有仪式感，内核的休闲循环越被当成"在玩游戏"而非"在用工具"——这正是 §1.2 补的"完整游戏包装"支柱。

### 17.2 店铺实景（世界）内部结构
- **视角**：室内 **45° 俯视**（类《模拟人生》/《双点》系列），真 3D 低多边形卡通。相机可轻量旋转 / 缩放（移动端双指），但**默认锁定 45°** 以避免迷失方向。
- **黑色阴影（光照揭示机制）**：
  - *氛围层*：用点光源 / 聚光灯界定"可玩区"，外围压暗到近黑，营造"黑暗中亮起的小店"的温馨与聚焦感。
  - *机制层（推荐作为可玩系统）*：**初始仅入口 + 柜台一带亮起；未被灯光覆盖处为黑色阴影**。随**店铺升级 / 完成任务 / 解锁成就**，灯光范围向外扩张，更多货架、后厨、二楼逐步"从黑暗中亮起"。这是强留存钩子（"我的店在长大"），且天然与任务 / 成就系统耦合（对照 §18 / §19）。
  - *设计假设（待验证）*：光照揭示到底是"纯美术氛围"还是"进度解锁机制"，需首轮原型验证玩家是否感知到"扩张"的快感；若无感，降级为纯美术（风险见 §17.5）。
- **HUD（浮于 3D 之上）**：店长等级 / 头像、金币💰、积分★、钻石💎、信誉条、设置齿轮⚙️。
- **底部功能坞（Dock）**：合成 / 订单 / 广告 / 商城 / 任务 / 成就 —— 以**模态层（Modal / Overlay）**形式在原 3D 场景上弹出，而非整屏切换，保持"我在店里操作"的临场感（**去 App 化关键**）。
- **设置按钮（⚙️）**：音效开关、音乐开关、语言、账号（绑定 / 解绑 / 退出登录）、重看新手引导。

### 17.3 资源加载策略（对照 §1.3 中端机）
- 首包只装"启动封面 + 账号 + 菜单 + 最小店铺区块"；店铺完整资产在 Loading 与**进入店铺后的后台流式加载**（UE5 Asset Manager / 异步加载 + 纹理流送）。
- 未覆盖区域（黑色阴影）在本阶段可直接用**暗场占位**，灯光扩张时再实例化对应资产，省首包与首帧。

### 17.4 与现有系统关系
- 账号 → token → TCP 鉴权（§14）；菜单 → 世界是唯一的玩法场景切换。
- 任务 / 成就（§18 / §19）的奖励与 §17.2 灯光揭示耦合，形成"目标 → 完成 → 奖励 → 揭示"正循环。

### 17.5 风险与回滚
- **光照揭示无感** → 降级为纯美术氛围，保留扩张动画作装饰。
- **45° 相机导致小模型点不中** → 关键操作（招募座位）仍走 UMG 确认气泡（§15.2）；合成核心手势为拖拽（§4.1），不依赖精准点 3D 模型。
- **定义"坏"**：Splash→Menu 流失率 > 40% → 开场过长，压缩 Loading / 合并封面与菜单。

## 18. 任务系统（Tasks）

### 18.1 分层
- **每日任务（3 条 / 日）**：合成 N 次 / 完成 N 单 / 看 N 个广告。每日重置，保底产出，承接日活（P5）。
- **每周任务（2–3 条 / 周）**：累计赚 X 金币 / 招募 N 名厨师 / 满好感 N 位顾客。承接周活。
- **章节任务（绑定 §15.4）**：每章一串经营目标，通关给章节奖励（已在 §8.2 建模）。

### 18.2 任务板交互
- 从功能坞「任务」打开模态层，分"每日 / 每周 / 章节"三页签。
- 完成态高亮，可一键「领取」→ 服务端发放（§14 HTTP `/task/claim`），防本地作弊。

## 19. 成就系统（Achievements）

### 19.1 设计意图
- 给"长期玩家"**里程碑式荣誉感**与**确定性目标**，补足留存长板（P5）；与 §17.2 灯光揭示、§18 任务形成"目标 → 完成 → 奖励 → 揭示"正循环。
- 分级：**铜 / 银 / 金**（同一成就随进度升阶，奖励递增）。

### 19.2 示例成就（首版草案）

| 成就 | 条件 | 分级奖励 |
|------|------|----------|
| 首次合成 | 完成第 1 次合成 | 铜：积分 ×50 |
| 店铺升级者 | 店铺达 Lv.5 / Lv.10 / Lv.20 | 银：钻石 ×1 / 金：钻石 ×3 |
| 招募大师 | 累计招募 10 / 50 / 100 名店员 | 铜→金：金币 / 钻石 |
| 满好感邻居 | 第 1 / 5 / 10 位顾客满好感 | 解锁专属剧情 + 积分 |
| 七日之约 | 连续登录 7 / 30 / 100 天 | 钻石 + 限定装饰 |
| 信誉楷模 | 信誉维持 100 达 7 天 | 限定装饰 |

### 19.3 交互
- 功能坞「成就」打开图鉴模态层，已解锁 / 进行中 / 未解锁三态；进度条显示当前阶进度。
- 解锁瞬间**全屏庆祝动效 + 称号飘字**（强正反馈），并可能触发 §17.2 灯光扩张（如"店铺升级者"解锁后后厨灯光亮起）。
- 奖励发放走服务端 `/achievement/claim`（§14）。

### 19.4 与现有系统关系
- 产出金币 / 积分 / 钻石 → 已建模于《经济系统平衡表》§9（成就奖励）与 §2.2（钻石来源）。
- 与任务互补：任务 = 短期节奏，成就 = 长期荣誉。
